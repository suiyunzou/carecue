# 问康 CareCue 产品需求文档

版本：v0.4  
日期：2026-06-09  
状态：阶段 3 & 4 核心交互机制设计与 Agent 后端实现已完成，长辈UI适老化优化及通用场景兜底已完成。

## 1. 项目概述

问康 CareCue 是一个面向长辈、子女和不熟悉搜索用户的就医前健康咨询产品。

产品目标不是替代医生诊断，而是在用户就医前帮助其完成三件事：

1. 把模糊、紧张、零散的症状描述整理清楚。
2. 基于追问、权威信息检索和风险规则给出初步方向与就诊建议。
3. 生成可带去医院、可转发给家人的结构化病情摘要。

## 2. 产品定位

### 2.1 一句话定位

问康是“就医前症状整理与日常健康咨询助手”。

### 2.2 产品边界

问康可以做：

- 症状信息追问与结构化整理
- 常见健康问题的初步解释
- 风险分级和急症提醒
- 建议就诊科室和就医准备材料
- 日常注意事项和随访建议
- 用户健康画像和历史咨询记忆

问康不做：

- 不给出最终确诊
- 不替代医生开药、停药或调整剂量
- 不承诺疾病判断准确率
- 不处理需要立即急救的场景
- 不在无证据时输出确定性医学结论

## 3. 背景与问题

### 3.1 用户痛点

长辈或不擅长搜索的用户常见问题：

- 不知道如何描述症状，只会说“不舒服”“疼”“晕”
- 就医时紧张，遗漏关键信息
- 子女远程沟通时无法快速判断严重程度
- 搜索引擎结果复杂，用户难以判断真假
- 同一个问题问多个 AI 得到不同回答，缺少综合判断机制
- 挂号前不知道该挂哪个科室

### 3.2 医生侧问题

医生接诊时需要快速获得：

- 主要症状
- 起病时间
- 变化过程
- 疼痛或不适部位
- 严重程度
- 伴随症状
- 既往病史
- 当前用药
- 过敏史
- 是否出现危险信号

如果这些信息缺失，医生只能现场反复询问，问诊效率降低，用户也容易遗漏事实。

## 4. 目标用户

### 4.1 主要用户

长辈用户：

- 年龄约 55 岁以上
- 手机使用能力中低
- 健康问题频率较高
- 对复杂搜索和专业医学术语不熟悉

子女用户：

- 帮父母远程判断是否需要就医
- 帮父母整理病情、预约挂号、陪诊
- 希望看到清晰、可信、可转发的信息

普通健康咨询用户：

- 想了解轻微不适可能原因
- 想知道是否需要马上去医院
- 想获得日常注意事项

### 4.2 次要用户

陪诊人员、家庭成员、基层健康管理人员。

## 5. 核心价值

### 5.1 对用户

- 用简单问题引导用户说清楚
- 降低搜索和医学理解门槛
- 遇到高危症状时及时提醒线下就医
- 帮助用户就诊前准备材料

### 5.2 对子女

- 快速理解长辈情况
- 判断是否需要陪同就医
- 获取可转发、可保存的病情摘要

### 5.3 对长期产品

- 形成家庭健康记忆
- 跟踪症状变化趋势
- 支持周度、月度健康建议
- 通过结构化数据持续优化追问质量

## 6. 核心功能拆解

本项目的核心功能不是“让 AI 直接回答健康问题”，而是围绕就医前决策形成一个可信流程：先把信息问清楚，再做风险分级，再给出下一步行动，最后沉淀为可复用健康记忆。

### 6.1 核心功能总表

| 核心功能 | 子功能 | 要实现的能力 | 预期结果 | MVP 优先级 |
| --- | --- | --- | --- | --- |
| 产品认知与转化 | 首页、产品说明、安全边界、立即体验 | 让用户理解问康能做什么、不能做什么，并进入咨询 | 用户知道产品不是确诊工具，愿意开始一次咨询 | P0 |
| 用户身份与数据隔离 | 登录、登出、用户会话、记录权限 | 确保健康记录只属于当前用户 | 未登录不能看记录，已登录只能看自己的记录 | P0 |
| 症状输入 | 自然语言输入、示例问题、患者基础信息补齐、紧急提示 | 低门槛收集用户最初问题 | 用户能用口语描述“不舒服”“头晕”等模糊问题 | P0 |
| 智能追问 | 主诉识别、基础信息补齐、症状细节追问、跳过/不清楚 | 把模糊问题转成医生可用的信息 | 关键字段补齐前不直接输出判断 | P0 |
| 红旗风险识别 | 高危症状规则、急症提示、流程中断、急诊摘要 | 遇到危险信号时优先提醒线下急诊 | 高危情况第一时间被明确提示 | P0 |
| 初步判断与就诊建议 | 紧急程度、可能方向、科室建议、注意事项、不确定项 | 给用户一个清晰、克制、可执行的下一步 | 用户知道是否急诊、门诊、观察，以及该挂什么科 | P0 |
| 医生沟通摘要 | 摘要生成、复制、下载/转发预留 | 将咨询信息整理成就诊沟通材料 | 用户到医院时可以快速说清病情 | P0 |
| 历史记录 | 咨询列表、详情、删除 | 保存单次咨询闭环 | 用户能回看过往咨询，不丢失信息 | P1 |
| 健康画像与记忆 | 常见症状、慢病、用药、过敏、家人档案、记忆开关 | 让新咨询可以结合授权历史信息 | 系统能更准确地追问和提醒 | P1 |
| 权威检索 | 来源白名单、联网查询、引用展示、检索失败处理 | 用可靠来源约束 AI 回答 | 关键建议有依据，不伪造来源 | P1 |
| 多模型交叉验证 | 多模型并行、共识/分歧、置信度、人工复核入口 | 降低单一模型不确定性 | 输出共识项，冲突项降权提示 | P2 |
| 周/月健康建议 | 周报、月报、趋势提醒、就医复盘 | 从单次咨询升级为长期健康管理 | 用户获得周期性健康建议 | P2 |

优先级定义：

- P0：MVP 必须具备，否则无法形成就医前咨询闭环。
- P1：上线前建议具备，影响可信度、留存和数据价值。
- P2：增长和差异化功能，可在核心闭环稳定后开发。

### 6.2 已确认产品决策

| 决策项 | 已确认方向 | 产品原因 |
| --- | --- | --- |
| 登录能力 | 阶段 2 开始做最小可用真实登录，不做纯前端模拟登录 | 登录和数据隔离是核心能力，后续要上线服务器，不能只停留在演示态 |
| 数据存储 | 阶段 2 开始接入数据库，不使用浏览器 localStorage 保存核心业务数据 | 历史咨询、用户记录和健康画像都需要服务端持久化 |
| 数据库 | 优先选 PostgreSQL | 适合结构化用户数据、咨询记录、JSON 症状结构和后续权限扩展 |
| 第一批咨询场景 | 头晕、胸痛、咳嗽，以及泛化的综合健康咨询场景 | 覆盖常见咨询、高风险识别、慢性症状追问，并支持真实泛化场景兜底 |
| 本人/家人入口 | 阶段 2 不强制做成独立入口 | 用户核心动作是登录后描述问题；患者身份、年龄等信息在追问中补齐 |
| 追问交互 | 选择按钮优先，保留“其他/补充说明”文本输入 | 降低长辈输入压力，同时允许用户补充特殊情况 |

### 6.3 P0 核心闭环

P0 必须跑通以下链路：

```text
首页 -> 登录 -> 症状输入 -> 追问 -> 风险识别 -> 初步判断 -> 医生沟通摘要 -> 保存记录
```

每个环节的产品要求：

| 环节 | 输入 | 系统处理 | 输出 | 验收口径 |
| --- | --- | --- | --- | --- |
| 首页 | 用户访问网站 | 展示定位、价值、安全边界 | 用户点击立即体验 | 首屏能说明产品用途，不承诺确诊 |
| 登录 | 手机号/邮箱和密码 | 创建或验证用户身份，建立会话 | 进入咨询页 | 用户信息写入数据库，未登录无法进入历史记录 |
| 症状输入 | 用户自然语言描述 | 识别咨询对象和主诉 | 生成追问计划 | 输入“头晕”不能直接给疾病结论 |
| 追问 | 用户逐步回答 | 补齐年龄、时间、程度、伴随症状、病史等 | 结构化症状信息 | 每轮最多 1-3 个问题，可选“不清楚” |
| 风险识别 | 结构化症状和回答 | 匹配红旗规则 | 风险等级 A-D | 胸痛伴呼吸困难必须提示急诊 |
| 初步判断 | 已补齐信息、风险等级、检索依据 | 生成可能方向和行动建议 | 结果页 | 先显示紧急程度，再显示可能方向 |
| 医生摘要 | 用户回答和判断结果 | 按问诊习惯重组 | 可复制摘要 | 摘要客观，不写“确诊” |
| 保存记录 | 当前咨询内容 | 写入 PostgreSQL | 历史详情 | 记录归属当前用户，可删除 |

### 6.4 产品差异化能力

为了避免未来被通用大模型能力替代，问康需要持续强化以下差异化：

- 追问流程产品化：不是一次性聊天，而是按就医逻辑补齐关键字段。
- 红旗症状规则化：高危场景不依赖模型自由发挥。
- 权威来源约束：结果能看到依据，检索失败时明确说明。
- 医生摘要沉淀：输出能直接用于挂号、急诊、陪诊和家庭沟通。
- 家庭健康记忆：围绕本人和家人形成长期画像，而不是孤立对话。
- 多模型共识机制：把多个模型的分歧转化为可解释的风险提示。

## 7. MVP 范围

MVP 先做网页端，完成从首页到咨询结果的闭环。

### 7.1 MVP 包含

- 首页
- 最小可用真实登录
- PostgreSQL 数据库存储
- 咨询入口
- 症状输入
- 规则/AI 追问流程
- 红旗症状提醒
- 初步判断结果页
- 建议就诊科室
- 日常注意事项
- 医生沟通摘要
- 历史咨询记录列表

### 7.2 MVP 不包含

- 在线问医生
- 真实挂号支付
- 处方建议
- 医保、医院系统对接
- 医疗影像识别
- 检验报告自动解读
- 多模型实时并行调用

## 8. 页面与功能需求

### 8.1 首页

目标：让用户理解产品是什么，并进入体验。

页面内容：

- 品牌名称：问康 CareCue
- 主标题：就医前，先把症状说清楚
- 产品说明：通过追问、联网查证和风险分级，帮助用户理清症状、判断紧急程度、准备沟通材料
- 主按钮：立即体验
- 次按钮：先看安全边界
- 安全提示：问康不是确诊工具，出现高危症状时应立即联系急救或线下就医
- 工作方式：先追问、再判断、再整理
- 可信机制：权威来源、红旗规则、低置信度提示
- 技术方向：当前 Demo、服务端阶段、长期能力

验收标准：

- 1440px 桌面端首屏展示品牌、主标题、说明、CTA 和安全提示
- 390px 移动端不出现横向滚动
- 所有主要按钮触控区域不小于 44px
- 页面文案不承诺确诊

### 8.2 登录

目标：确保用户只能查看自己的健康记录。

MVP 方案：

- 支持手机号/邮箱 + 密码的最小可用登录
- 用户信息写入数据库
- 密码必须哈希存储，不能明文保存
- 登录后建立服务端会话或签发访问令牌
- 登录后进入咨询首页
- 未登录点击“立即体验”时跳转登录页

后续方案：

- 接入手机号验证码、微信登录或第三方 Auth
- 支持家庭成员档案切换
- 支持找回密码、修改手机号/邮箱

最小用户表字段：

| 字段 | 说明 | 阶段 |
| --- | --- | --- |
| id | 用户唯一 ID | 阶段 2 |
| phone | 手机号，可为空 | 阶段 2 |
| email | 邮箱，可为空 | 阶段 2 |
| password_hash | 密码哈希 | 阶段 2 |
| display_name | 昵称 | 阶段 2 |
| created_at | 创建时间 | 阶段 2 |
| updated_at | 更新时间 | 阶段 2 |
| last_login_at | 最近登录时间 | 阶段 2 |
| status | 正常、禁用、注销 | 阶段 5 |

验收标准：

- 未登录用户无法查看历史咨询
- 已登录用户只能看到自己的记录
- 数据库中能看到用户记录和最近登录时间
- 数据库中不能保存明文密码
- 登出后回到首页或登录页
- 注册成功后必须设置 httpOnly Cookie，会话可继续访问受保护接口
- 登录请求只依赖账号和密码；前端状态中出现空昵称字段时不应影响登录

### 8.3 咨询入口

目标：用低门槛方式收集用户初始问题。

页面元素：

- 大号输入框
- 示例问题
- 引导式补充问题
- 选择按钮
- “其他/补充说明”文本输入
- 语音输入预留入口
- 紧急情况提示

示例问题：

- 我爸最近总是头晕，是怎么回事？
- 胸口闷，需要马上去医院吗？
- 咳嗽两周不好，要挂什么科？
- 老人晚上腿抽筋，平时要注意什么？

验收标准：

- 用户能输入自然语言症状
- 系统能在追问中补齐患者年龄、性别、既往病史等基础信息
- 用户输入过短时，引导补充信息

### 8.4 追问流程

目标：防止用户描述模糊导致回答不准确。

追问策略：

- 先识别主诉
- 再补齐基础信息
- 再补齐症状细节
- 再检查红旗症状
- 最后确认是否有既往病史、用药和过敏史

基础追问字段：

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| 咨询对象 | 本人或家人 | 是 |
| 年龄 | 年龄段或具体年龄 | 是 |
| 性别 | 男、女、其他/不便透露 | 否 |
| 主诉 | 用户最主要不适 | 是 |
| 起病时间 | 什么时候开始 | 是 |
| 持续时间 | 持续多久 | 是 |
| 严重程度 | 轻、中、重或 0-10 分 | 是 |
| 伴随症状 | 发热、呕吐、胸闷等 | 是 |
| 既往病史 | 高血压、糖尿病、心脏病等 | 否 |
| 当前用药 | 正在吃的药 | 否 |
| 过敏史 | 药物或食物过敏 | 否 |

追问交互要求：

- 每次最多问 1-3 个问题
- 优先使用选择题，其次是短文本
- 允许用户选择“不清楚”
- 每组选择题提供“其他/补充说明”
- 支持跳过非关键问题
- 对长辈使用口语化表达
- 患者是本人还是家人不作为第一版独立入口，但可以在追问中询问“这次不舒服的人是谁、大概多大年纪”

验收标准：

- 用户输入“头晕”时，系统必须追问年龄、持续时间、是否伴随肢体无力/说话不清/胸闷等危险信号
- 用户输入“胸痛”时，系统必须优先询问持续时间、疼痛性质、是否出汗/呼吸困难/放射痛，并展示急症风险提示
- 未补齐关键字段前，不直接输出确定性疾病判断

### 8.5 红旗症状提醒

目标：将可能危及生命或需要立即就医的情况前置。

高危提示示例：

- 胸痛伴呼吸困难、出汗、放射到左臂或下颌
- 突发剧烈头痛
- 口角歪斜、肢体无力、说话不清
- 意识模糊或昏迷
- 严重过敏反应
- 持续高热伴精神差
- 呕血、黑便、大量出血
- 老人跌倒后头痛、呕吐或意识变化

产品行为：

- 出现红旗症状时，中断普通咨询流程
- 展示明显急症提示
- 建议拨打当地急救电话或立即线下就医
- 仍可生成“急诊沟通摘要”

验收标准：

- 高危提示必须在结果页第一屏展示
- 不能仅在长文本中弱提示
- 不能用“可能没事”弱化风险

### 8.6 初步判断结果页

目标：清晰表达风险、可能方向和下一步行动。

结果页结构：

1. 紧急程度
2. 可能方向
3. 建议就诊科室
4. 日常注意事项
5. 需要继续观察的变化
6. 医生沟通摘要
7. 依据和不确定项

紧急程度分级：

| 等级 | 名称 | 产品表达 |
| --- | --- | --- |
| A | 立即就医 | 建议立即急诊或联系急救 |
| B | 尽快就医 | 建议 24-48 小时内就医 |
| C | 门诊评估 | 可预约相关科室门诊 |
| D | 日常观察 | 暂无明显高危信号，注意观察变化 |

输出要求：

- 不能写“你得了某某病”
- 应写“可能与以下方向有关”
- 每个可能方向必须说明支持点和不支持点
- 不确定信息要明确列出

验收标准：

- 结果页必须先显示紧急程度，再显示可能方向
- 每条建议都能追溯到用户回答或检索依据
- 低置信度时必须提醒线下就医或补充信息

### 8.7 医生沟通摘要

目标：帮助用户在医院快速说清楚。

摘要格式：

```text
就诊前病情摘要

咨询对象：父亲，约 68 岁，男
主要不适：近 3 天反复头晕
起病时间：2026-06-05 左右
症状特点：站起时更明显，休息后缓解
伴随症状：无胸痛，无说话不清，无肢体无力
既往病史：高血压 10 年
当前用药：降压药，具体名称不清楚
需要医生重点评估：血压波动、脑血管风险、耳源性眩晕等
```

功能要求：

- 支持复制
- 支持下载为文本或 PDF
- 支持转发给家人
- 急症场景生成急诊版摘要

验收标准：

- 摘要不包含 AI 夸大判断
- 摘要按医生问诊习惯排序
- 用户可一键复制

## 9. AI 与检索策略

### 9.1 总体原则

AI 不是单独决策者，而是信息整理、追问生成和摘要生成工具。

系统判断由四部分组成：

1. 用户输入和追问回答
2. 红旗症状规则
3. 权威来源检索
4. AI 结构化分析和自检

### 9.2 权威来源白名单

优先来源：

- 国家卫健委
- 国家药监局
- 三甲医院官方科普
- 医学会/专业指南
- WHO
- CDC
- NHS

来源使用规则：

- 不使用无出处的营销号内容
- 不使用无法确认作者和机构的医疗文章
- 不把单一网页作为唯一依据
- 重要风险建议至少需要规则或权威来源支持

### 9.3 多模型交叉验证

MVP：

- 单模型结构化输出
- 检索证据约束
- 红旗规则兜底
- 自检提示词

后续：

- 多模型并行给出可能方向
- 计算一致性
- 对冲突结论降权
- 输出“共识项”和“分歧项”

### 9.4 防幻觉机制

要求：

- 使用 JSON Schema 约束输出结构
- 回答必须包含不确定项
- 无依据结论不得展示
- 医疗建议必须区分“建议观察”“建议门诊”“建议急诊”
- 对药物相关问题默认提示咨询医生或药师

## 10. 数据与隐私

医疗健康信息属于敏感个人信息，应按敏感个人信息处理。

要求：

- 明示收集目的
- 获得用户单独同意
- 最小化收集
- 用户可查看、导出、删除个人记录
- 默认不公开、不共享
- 日志中避免记录完整症状原文
- 生产环境加密存储关键健康数据
- 管理后台访问需要权限控制和审计

未成年人数据：

- 需要监护人同意
- 默认不面向未成年人独立使用

### 阶段 3：真实 AI 接入

阶段目标：将固定规则原型升级为 AI 驱动，但仍用结构化约束保证稳定和安全。

详细开发文档见：[阶段 3 AI 分析接入开发文档](STAGE3_AI_DEVELOPMENT.md)。

核心功能：

| 核心功能 | 子功能 | 实现要求 | 预期结果 |

| 追问生成 | 基于用户输入生成下一轮问题 | 使用 JSON Schema 限定问题结构 | AI 问题可渲染为稳定 UI |
| 结果生成 | 紧急程度、可能方向、建议科室、注意事项 | 输出必须符合固定结果结构 | 结果页不会因为模型自由发挥而失控 |
| 自检机制 | 确诊措辞检查、字段缺失检查、红旗复核 | 输出前进行二次校验 | 降低错误和过度确定性表达 |
| Prompt 管理 | Prompt 版本号、场景配置、回滚 | 每次调整可追踪 | 方便测试不同策略效果 |

- AI 输出 95% 以上能通过结构化解析；解析失败时展示可恢复错误。
- 缺少年龄、持续时间、严重程度等关键字段时，AI 必须先追问。
- 输出中不能出现“你已经确诊为”“一定是”等确定性诊断措辞。
- 红旗规则优先级高于 AI 普通判断。
- 模型超时或失败时，用户不会丢失已填写内容。

暂不包含：

- 多模型并行。
- 自动诊断准确率承诺。
- 药物剂量、处方、停药建议。

### 阶段 4：联网检索

阶段目标：让关键医学建议有来源约束，提升可信度并降低 AI 幻觉。

核心功能：

| 核心功能 | 子功能 | 实现要求 | 预期结果 |
| --- | --- | --- | --- |
| 来源白名单 | 官方机构、医院、指南、权威科普 | 配置可维护的来源白名单和黑名单 | 避免营销号进入判断依据 |
| 搜索策略 | 按症状、风险、科室关键词检索 | 先搜索，再抽取摘要，再给 AI 使用 | AI 回答被证据约束 |
| 引用展示 | 来源标题、机构、链接、摘要 | 结果页展示依据来源 | 用户能判断建议从哪里来 |
| 证据冲突处理 | 多来源不一致、证据不足 | 标记为“不确定”或建议线下就医 | 不把弱证据包装成强结论 |
| 检索失败兜底 | 超时、无结果、来源不可用 | 明确说明检索失败，不伪造引用 | 保持可信度 |

验收标准：

- 结果页中涉及就医建议、红旗风险、常见病方向时展示至少 1 个依据来源或规则依据。
- 非白名单来源不得进入核心判断依据。
- 检索失败时显示“本次未能完成联网查证”，不能生成假链接。
- 来源摘要不能替代原文，必须保留来源链接。
- 用户可以展开查看“依据与不确定项”。

暂不包含：

- 医学论文深度综述。
- 实时医院号源查询。
- 医保、医院系统对接。

### 阶段 5：用户系统增强与健康档案

阶段目标：从一次性咨询升级为个人/家庭长期健康记录。

核心功能：

| 核心功能 | 子功能 | 实现要求 | 预期结果 |
| --- | --- | --- | --- |
| 登录增强 | 手机验证码、微信登录、找回密码、账号注销 | 在阶段 2 最小登录基础上增强 | 降低真实用户登录门槛 |
| 用户数据隔离增强 | 用户 ID、记录权限、服务端鉴权、审计 | 后端按用户身份查询记录 | 任何用户不能越权查看他人记录 |
| 历史咨询 | 列表、详情、搜索、删除 | 保存结构化问答、结果和摘要 | 用户能复盘历史咨询 |
| 家庭成员档案 | 本人、父亲、母亲、配偶等 | 每次咨询选择对象 | 子女代问场景更清晰 |
| 健康画像 | 慢病、用药、过敏、常见症状 | 从用户确认的信息中沉淀 | 新咨询能更准确追问 |
| 记忆开关 | 开启/关闭、删除记忆 | 用户控制历史信息是否参与新对话 | 满足隐私和信任要求 |
| 数据导出删除 | 导出记录、删除单条/全部 | 用户可管理个人数据 | 满足基本数据权利要求 |

验收标准：

- 用户只能看到自己的咨询记录和家庭成员档案。
- 新对话可以读取用户授权的慢病、用药、过敏信息。
- 用户关闭记忆后，新对话不再使用历史画像。
- 用户删除记录后，历史列表不再展示该记录。
- 登录态失效后无法访问记录详情。

暂不包含：

- 医生端后台。
- 家庭成员邀请协作。
- 医院电子病历同步。

### 阶段 6：多模型验证与健康建议

阶段目标：提升判断稳定性，形成问康相对于通用 AI 的长期壁垒。

核心功能：

| 核心功能 | 子功能 | 实现要求 | 预期结果 |
| --- | --- | --- | --- |
| 多模型交叉验证 | 主模型、复核模型、规则模型 | 对关键结论进行一致性比较 | 降低单一模型偏差 |
| 共识与分歧展示 | 共识项、冲突项、置信度 | 分歧较大时降级为“需要线下评估” | 用户知道哪些结论更可靠 |
| 周度建议 | 本周咨询主题、重复症状、风险提醒 | 基于历史记录生成短建议 | 用户了解近期健康变化 |
| 月度建议 | 趋势、慢病关注、就医复盘 | 输出更完整的月度总结 | 产品从咨询工具变成健康陪伴 |
| 人工复核预留 | 高风险样本、低置信度样本 | 后台标记待复核记录 | 为未来医疗专业合作预留空间 |

验收标准：

- 多模型输出不一致时，结果页展示“分歧项”，而不是强行给单一结论。
- 周报/月报不得制造疾病焦虑，重点放在观察、复诊准备和生活方式建议。
- 高风险记录可以被标记为需要人工复核。
- 用户可以关闭周期性建议。

暂不包含：

- 医生在线诊疗。
- 付费问诊。
- 自动生成处方或治疗方案。

### 14.1 阶段优先级建议

建议开发顺序：

1. 阶段 2：咨询流程原型。
2. 阶段 3：真实 AI 接入。
3. 阶段 4：联网检索。
4. 阶段 5：用户系统与健康档案。
5. 阶段 6：多模型验证与健康建议。

原因：

- 咨询流程是产品核心，必须先验证用户是否能顺利完成。
- AI 接入前先用规则原型锁定交互和结果结构，能减少后续返工。
- 联网检索和多模型验证都依赖稳定的结果结构。
- 健康画像需要先有足够稳定的咨询数据结构。

## 15. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AI 输出错误 | 用户误判健康风险 | 红旗规则、检索依据、低置信度提示 |
| 用户把结果当确诊 | 医疗安全风险 | 明确产品边界，禁止确诊措辞 |
| 医疗健康数据泄露 | 严重合规风险 | 最小化收集、加密、权限控制 |
| 追问过长 | 用户流失 | 每轮 1-3 个问题，允许跳过 |
| 老年用户不会操作 | 无法完成咨询 | 大字、少按钮、语音输入预留 |
| 大模型能力被平台内置 | 产品可替代性增强 | 做健康记忆、家庭档案、医生摘要和可信证据 |
