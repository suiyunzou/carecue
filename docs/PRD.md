# 问康 CareCue 产品需求文档

版本：v1.0
日期：2026-06-10
状态：阶段 3 & 4 核心交互机制设计与 Agent 后端实现已完成，长辈UI适老化优化及通用场景兜底已完成。

## 1. 项目概述

问康 CareCue 是一个面向长辈、子女和不熟悉搜索用户的就医前健康咨询产品。

产品目标不是替代医生诊断，而是在用户就医前帮助其完成三件事：

1. 把模糊、紧张、零散的症状描述整理成结构化信息。
2. 基于追问、权威信息检索和风险规则给出初步方向与就诊建议。
3. 生成可带去医院、可转发给家人的结构化病情摘要。

### 1.1 一句话定位

问康是"就医前症状整理与日常健康咨询助手"。

### 1.2 产品边界

**可以做：**
- 症状信息追问与结构化整理
- 常见健康问题的初步解释
- 风险分级和急症提醒
- 建议就诊科室和就医准备材料
- 日常注意事项和随访建议
- 用户健康画像和历史咨询记忆
- 支持真实场景的泛化健康咨询（任意主诉均可触发通用追问）

**不做：**
- 不给出最终确诊
- 不替代医生开药、停药或调整剂量
- 不承诺疾病判断准确率
- 不处理需要立即急救的场景
- 不在无证据时输出确定性医学结论

## 2. 背景与问题

### 2.1 用户痛点

长辈或不擅长搜索的用户常见问题：
- 不知道如何描述症状，只会说"不舒服""疼""晕"
- 就医时紧张，遗漏关键信息
- 子女远程沟通时无法快速判断严重程度
- 搜索引擎结果复杂，用户难以判断真假
- 同一个问题问多个 AI 得到不同回答，缺少综合判断机制
- 挂号前不知道该挂哪个科室

### 2.2 医生侧问题

医生接诊需要快速获得：主要症状、起病时间、变化过程、疼痛部位、严重程度、伴随症状、既往病史、当前用药、过敏史、危险信号。

如果这些信息缺失，医生只能现场反复询问，问诊效率降低。

## 3. 目标用户

| 角色 | 描述 | 核心诉求 |
| --- | --- | --- |
| 长辈用户 | 55岁以上，手机使用能力中低，健康问题频率较高 | 简单描述不适，获得可靠的就医方向 |
| 子女用户 | 帮父母远程判断是否需要就医、整理病情 | 快速理解情况，获取可转发的病情摘要 |
| 普通健康咨询用户 | 想了解轻微不适可能原因和是否需要就医 | 低门槛输入，获得克制且有依据的建议 |

次要用户：陪诊人员、家庭成员、基层健康管理人员。

## 4. 核心价值

**对用户：** 用简单问题引导说清症状；降低搜索和医学理解门槛；高危症状及时提醒就医；帮助就诊前准备材料。

**对子女：** 快速理解长辈情况；判断是否需要陪同就医；获取可转发、可保存的病情摘要。

**对长期产品：** 形成家庭健康记忆；跟踪症状变化趋势；支持周期性健康建议；通过结构化数据持续优化追问质量。

## 6. 已确认产品决策

| 决策项 | 已确认方向 | 原因 |
| --- | --- | --- |
| 登录能力 | 最小可用真实登录，不做纯前端模拟登录 | 数据隔离是核心能力 |
| 数据存储 | PostgreSQL，不使用浏览器 localStorage 保存核心业务数据 | 历史咨询、用户记录需服务端持久化 |
| 数据库 | PostgreSQL | 适合结构化用户数据、咨询记录和 JSON 症状结构 |
| 咨询场景 | 头晕、胸痛、咳嗽，以及泛化的综合健康咨询场景 | 覆盖常见咨询、高风险识别、通用兜底 |
| 本人/家人入口 | 不强制做成独立入口 | 患者身份、年龄等信息在追问中补齐 |
| 追问交互 | 选择按钮优先，保留"其他/补充说明"文本输入 | 降低长辈输入压力，允许补充特殊情况 |
| 技术架构 | 前后端分离：React + TypeScript + Vite / Express + Prisma + PostgreSQL | 前后端独立迭代，服务端持久化 |

## 7. 页面与功能需求

### 7.1 首页

目标：让用户理解产品是什么，并进入体验。

- 品牌名称：问康 CareCue
- 主标题：就医前，先把症状说清楚
- 产品说明：通过追问、联网查证和风险分级，帮助用户理清症状、判断紧急程度、准备沟通材料
- 主按钮：立即体验（触控区域 ≥ 44px）
- 次按钮：先看安全边界
- 安全提示：问康不是确诊工具，出现高危症状时应立即联系急救或线下就医

**验收标准：**
- 1440px 桌面端首屏展示品牌、主标题、说明、CTA 和安全提示
- 390px 移动端不出现横向滚动
- 所有主要按钮触控区域不小于 44px
- 页面文案不承诺确诊

### 7.2 登录

目标：确保用户只能查看自己的健康记录。

MVP 方案：手机号/邮箱 + 密码的最小可用登录。用户信息写入数据库，密码哈希存储，登录后建立服务端会话或签发访问令牌。未登录点击"立即体验"时跳转登录页。

后续方案：手机号验证码、微信登录、家庭成员档案切换、找回密码。

**最小用户表字段：** id、phone、email、password_hash、display_name、created_at、updated_at、last_login_at、status

**验收标准：**
- 未登录用户无法查看历史咨询
- 已登录用户只能看到自己的记录
- 数据库中不能保存明文密码
- 注册成功后 httpOnly Cookie 可继续访问受保护接口
- 登录请求只依赖账号和密码，空昵称字段不影响登录

### 7.3 咨询入口（适老化优化）

目标：用低门槛方式收集用户初始问题，通过适老化与情感化设计降低心理障碍。

**页面元素：** 输入框、示例问题、引导式补充问题、选择按钮、"其他/补充说明"文本输入、语音输入预留入口、紧急情况提示。

**适老化设计要求：**
- **主标题区域：** 字号适度（text-2xl md:text-3xl），行高和字重保持阅读舒适，避免大面积粗体造成压迫感。
- **输入区域：** 输入框中等高度（rows=3~4，min-h-[120px]），提示语文案口语化（如"比如：我爸最近总头晕，站起来更明显。"），避免让用户误以为需要撰写长篇大论。
- **紧急提醒卡片：** 默认状态采用浅色背景（bg-gray-50/slate-50）、柔和边框（border-gray-200）和温和图标，仅当检测到高危词汇时动态过渡为强警示状态（bg-orange-50 border-orange-200 text-orange-800）。
- **示例按钮：** 严格保留，点击即输入，确保文本正确填入输入框并可触发后续流程。
- **主按钮文案：** 维持"开始追问"，比"发送"或"生成报告"更符合阶段 3 产品逻辑，触控区域 ≥ 44px。

**示例问题：**
- 我爸最近总是头晕，是怎么回事？
- 胸口闷，需要马上去医院吗？
- 咳嗽两周不好，要挂什么科？
- 老人晚上腿抽筋，平时要注意什么？

**泛化场景支持：**
- 系统已支持真实场景输入，不仅限于头晕、胸痛、咳嗽三类预设场景。
- 任意非预设症状均触发通用综合健康咨询（general）场景兜底，按通用追问模板补齐信息。
- 前端引导文案体现对真实多样化输入的支持能力。

**验收标准：**
- 用户能输入自然语言症状
- 页面整体视觉柔和，无过度夸张的字体与色块
- Placeholder 为指定口语化文案，输入框高度中等
- 右侧安全提醒卡在普通状态下不刺眼，输入红旗词汇后能实时切换为警示状态
- 任意非预设症状能触发通用追问流程
- 示例按钮点击交互正常，文本正确填入输入框

### 7.4 追问流程

目标：防止用户描述模糊导致回答不准确。

**追问策略：** 先识别主诉 → 再补齐基础信息 → 再补齐症状细节 → 再检查红旗症状 → 最后确认既往病史、用药和过敏史。

**基础追问字段：**

| 字段 | 说明 | 是否必填 |
| --- | --- | --- |
| 咨询对象 | 本人或家人 | 是 |
| 年龄 | 年龄段或具体年龄 | 是 |
| 性别 | 男、女、其他/不便透露 | 否 |
| 主诉 | 用户最主要不适 | 是 |
| 起病时间 | 什么时候开始 | 是 |
| 持续时间 | 持续多久 | 是 |
| 严重程度 | 轻、中、重或 0-10 分 | 是 |
| 伴随症状 | 发热、呕吐、胸闷等 | 是 |
| 既往病史 | 高血压、糖尿病、心脏病等 | 否 |
| 当前用药 | 正在吃的药 | 否 |
| 过敏史 | 药物或食物过敏 | 否 |

**追问交互要求：**
- 每次最多问 1-3 个问题
- 优先使用选择题，其次是短文本
- 允许用户选择"不清楚"
- 每组选择题提供"其他/补充说明"
- 支持跳过非关键问题
- 对长辈使用口语化表达
- 追问最多 1-3 轮

**通用场景兜底：**
- 泛化追问覆盖年龄、咨询对象、症状出现时间、通用红旗信号、伴随症状、既往病史和用药。
- 综合场景输出科室建议（全科医学科、急诊科）和方向提示，提供克制的日常观察建议。

**验收标准：**
- 用户输入"头晕"时追问年龄、持续时间、是否伴随肢体无力/说话不清/胸闷
- 用户输入"胸痛"时优先询问持续时间、疼痛性质、是否出汗/呼吸困难/放射痛，并展示急症风险提示
- 任意非预设症状（如"胃有点不舒服"）触发通用追问，检测到高危选项时正确升级为急诊提醒
- 未补齐关键字段前不直接输出确定性疾病判断

### 7.5 红旗症状提醒

目标：将可能危及生命或需要立即就医的情况前置。

**高危提示示例：**
- 胸痛伴呼吸困难、出汗、放射到左臂或下颌
- 突发剧烈头痛
- 口角歪斜、肢体无力、说话不清
- 意识模糊或昏迷
- 严重过敏反应
- 持续高热伴精神差
- 呕血、黑便、大量出血
- 老人跌倒后头痛、呕吐或意识变化

**产品行为：**
- 出现红旗症状时中断普通咨询流程
- 展示明显急症提示
- 建议拨打当地急救电话或立即线下就医
- 仍可生成"急诊沟通摘要"
- 前端动态强化：输入框内容匹配红旗词汇时，安全提醒卡片平滑过渡为强警示状态

**验收标准：**
- 高危提示必须在结果页第一屏展示
- 不能仅在长文本中弱提示
- 不能用"可能没事"弱化风险
- 普通场景安全提醒卡视觉柔和，高危场景实时强化

### 7.6 初步判断结果页

目标：清晰表达风险、可能方向和下一步行动。

**结果页结构：**
1. 紧急程度（优先级最高）
2. 可能方向（不超过 3 个，各有支持点和注意点）
3. 建议就诊科室
4. 日常注意事项
5. 需要继续观察的变化
6. 医生沟通摘要
7. 依据和不确定项

**紧急程度分级：**

| 等级 | 名称 | 产品表达 |
| --- | --- | --- |
| A | 立即就医 | 建议立即急诊或联系急救 |
| B | 尽快就医 | 建议 24-48 小时内就医 |
| C | 门诊评估 | 可预约相关科室门诊 |
| D | 日常观察 | 暂无明显高危信号，注意观察变化 |

**输出要求：**
- 不能写"你得了某某病"，应写"可能与以下方向有关"
- 每个可能方向必须说明支持点和不支持点
- 不确定信息要明确列出

**验收标准：**
- 结果页必须先显示紧急程度，再显示可能方向
- 每条建议都能追溯到用户回答或检索依据
- 低置信度时必须提醒线下就医或补充信息

### 7.7 医生沟通摘要

目标：帮助用户在医院快速说清楚。

**摘要格式示例：**
```
就诊前病情摘要

咨询对象：父亲，约 68 岁，男
主要不适：近 3 天反复头晕
起病时间：2026-06-05 左右
症状特点：站起时更明显，休息后缓解
伴随症状：无胸痛，无说话不清，无肢体无力
既往病史：高血压 10 年
当前用药：降压药，具体名称不清楚
需要医生重点评估：血压波动、脑血管风险、耳源性眩晕等
```

**功能要求：** 支持复制、下载为文本或 PDF（后续）、转发给家人、急症场景生成急诊版摘要。

**验收标准：**
- 摘要不包含 AI 夸大判断
- 摘要按医生问诊习惯排序
- 用户可一键复制

### 7.8 历史记录与健康画像

**MVP：** 展示历史咨询列表、可查看单次详情、可删除记录。

**后续版本：** 周度健康总结、月度健康趋势、重复症状提醒、用药和慢病标签、家庭成员档案。

**健康画像字段：** 年龄段、性别、常见症状、慢病史、用药史、过敏史、最近咨询主题、高风险提醒历史。

**验收标准：**
- 新对话能读取用户授权范围内的历史画像
- 用户可关闭记忆功能
- 用户可删除单条记录或全部记录

## 8. AI 与检索策略

### 8.1 总体原则

AI 不是单独决策者，而是信息整理、追问生成和摘要生成工具。

系统判断由四部分组成：用户输入和追问回答、红旗症状规则、权威来源检索、AI 结构化分析和自检。

### 8.2 权威来源白名单

**优先来源：** 国家卫健委、国家药监局、三甲医院官方科普、医学会/专业指南、WHO、CDC、NHS。

**来源分级：**
- **A级：** 政府、疾控、WHO、NHS、默沙东、医学指南、三甲医院官网
- **B级：** 正规医疗平台、医生审核科普、丁香医生/丁香园
- **C级：** 普通健康资讯，仅弱参考
- **D级：** 论坛、自媒体、问答社区、营销医疗网站，默认不用

**使用规则：** 不使用营销号内容、不使用无法确认作者的医疗文章、不把单一网页作为唯一依据、重要风险建议至少需要规则或权威来源支持。

### 8.3 多模型交叉验证

**MVP：** 单模型结构化输出 + 检索证据约束 + 红旗规则兜底 + 自检提示词。

**后续：** 多模型并行给出可能方向 → 计算一致性 → 冲突结论降权 → 输出"共识项"和"分歧项"。

### 8.4 防幻觉机制

- 使用 JSON Schema 约束输出结构
- 回答必须包含不确定项
- 无依据结论不得展示
- 医疗建议必须区分"建议观察""建议门诊""建议急诊"
- 对药物相关问题默认提示咨询医生或药师

## 9. 数据与隐私

医疗健康信息属于敏感个人信息。

**要求：**
- 明示收集目的
- 获得用户单独同意
- 最小化收集
- 用户可查看、导出、删除个人记录
- 默认不公开、不共享
- 日志中避免记录完整症状原文
- 生产环境加密存储关键健康数据
- 管理后台访问需要权限控制和审计

**未成年人数据：** 需要监护人同意，默认不面向未成年人独立使用。

## 10. 分阶段功能规划

### 阶段 1：首页 Demo（已完成）

实现产品定位、视觉方向和第一入口。

### 阶段 2：咨询流程原型（已完成）

不用真实 AI，用规则和数据库跑通完整咨询闭环。已实现：真实注册/登录、JWT httpOnly Cookie 会话、PostgreSQL 持久化、头晕/胸痛/咳嗽/泛化综合场景规则追问、结果页、医生摘要、历史记录读写删除。

**质量门槛：** `npm run test:api`、`npm run test:agent`、`npm run typecheck:api`、`npm run lint`、`npm run build` 均需通过。

### 阶段 3：真实 AI 接入（已完成核心机制）

将固定规则原型升级为 AI 驱动，但用结构化约束保证稳定和安全。已实现 Agentic Workflow 核心代码结构（状态提取、检索路由、追问生成、报告生成）。

详见：[技术设计文档](TECHNICAL.md) 和 Agent 链路与 Prompt 设计章节。

**核心功能：** AI 网关、追问生成、结果生成、自检机制、Prompt 管理、反馈机制。

**验收标准：**
- AI 输出 95% 以上能通过结构化解析
- 缺关键字段时 AI 必须先追问
- 输出不能出现确诊措辞
- 红旗规则优先级高于 AI 普通判断
- 模型超时或失败时用户不丢失已填写内容

### 阶段 4：联网检索（已完成核心机制）

让关键医学建议有来源约束，提升可信度并降低 AI 幻觉。已实现来源白名单、Firecrawl 定向检索及决策路由机制。

**核心功能：** 来源白名单、搜索策略、引用展示、证据冲突处理、检索失败兜底。

**验收标准：**
- 结果页涉及就医建议时展示至少 1 个依据来源或规则依据
- 非白名单来源不得进入核心判断依据
- 检索失败时明确说明，不伪造引用
- 用户可以展开查看"依据与不确定项"

### 阶段 5：用户系统增强与健康档案（后续）

从一次性咨询升级为个人/家庭长期健康记录。

**核心功能：** 登录增强、用户数据隔离增强、家庭成员档案、健康画像、记忆开关、数据导出删除。

### 阶段 6：多模型验证与健康建议（后续）

提升判断稳定性，形成长期壁垒。

**核心功能：** 多模型交叉验证、共识与分歧展示、周度建议、月度建议、人工复核预留。

### 阶段优先级建议

阶段 2（原型） → 阶段 3（AI 接入） → 阶段 4（联网检索） → 阶段 5（用户系统增强） → 阶段 6（多模型验证）。

| 优先级 | 任务 | 目标 | 验收方式 |

## 11. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| AI 输出错误 | 用户误判健康风险 | 红旗规则、检索依据、低置信度提示 |
| 用户把结果当确诊 | 医疗安全风险 | 明确产品边界，禁止确诊措辞 |
| 医疗健康数据泄露 | 严重合规风险 | 最小化收集、加密、权限控制 |
| 追问过长 | 用户流失 | 每轮 1-3 个问题，允许跳过 |
| 老年用户不会操作 | 无法完成咨询 | 大字、少按钮、语音输入预留、适老化UI |
| 大模型能力被平台内置 | 产品可替代性增强 | 做健康记忆、家庭档案、医生摘要和可信证据 |

## 12. 关键指标

**MVP 指标：** 首页"立即体验"点击率、咨询流程完成率、平均追问轮次、结果页复制摘要率、高危场景提示触达率。

**长期指标：** 7 日留存、月活用户、家庭成员档案创建率、"有帮助"反馈比例、AI 回复不准确标记率、多模型分歧率。
