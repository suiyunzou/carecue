# 模块：M2 接入真实 LLM

> 对应《CareCue 4.0 设计文档》第三部分 M2：接入真实 LLM。
> 物理位置：`server/src/`（延续 M1，不改动遗留 `server/agent/`）。

## 职责

把 M1 的 Mock 决策替换为真实 tool-calling 模型（DeepSeek 主路径 + OpenRouter 回退），
补齐 Guard、可观测性与 HTTP 入口，并用 3 个真实症状端到端验证主循环对任意红旗知识库的通用性。

## 输入输出

- 决策层：`Llm.decide({ workspace, lastUserMessage, tools, feedback })` → `ToolCall`
  - `DeepSeekLlm`：底层 `chat.completions` 以 `CompleteFn` 注入（真实 OpenAI 客户端 / 测试假实现）
  - 基础设施级回退：当前 provider 整体失败（抛错/503）→ 切下一个 provider；**不在工具失败时回退**（设计文档 2.7）
- HTTP：`POST /api/consult { caseId, message, age?, sex? }` → `ConsultResponse`

## 依赖

- 内部：M1 全部模块 + `agent/trace`（可观测性）+ `routes/consult`
- 外部库：`openai`（OpenAI 兼容 tool calling）、`express`（路由）

## 核心逻辑

1. `DeepSeekLlm.decide` 把 `workspace.toSummary()` + 用户输入 + Guard/工具反馈拼成 user 消息，
   带 4 个工具的 OpenAI schema，`tool_choice: 'required'`，解析首个 `tool_call` 为 `ToolCall`。
2. provider 列表按 DeepSeek→OpenRouter 顺序尝试；每次成功/失败都写 `trace`（provider/model/tokens/duration/请求）。
3. `createConsultEngineFromEnv()`：配了 Key 用 `DeepSeekLlm`，否则回退 `MockLlm`（本地零配置可跑）。
4. Guard 防空转：红旗已加载时拒绝重复 `lookup_red_flags`，避免真实 LLM 反复检索导致无进展循环。
5. `MemoryTracer` 记录 decision / guard / tool / llm / snapshot，对齐 M3 的 PG 落库。

## 测试用例（Few Shot）

`npm run test:m2`（9 用例全过）：
1. 头痛逐轮排除 → 非确诊报告（Mock 驱动）
2. 腹痛命中急腹症 → emergency（Mock 驱动）
3. 头晕胸闷逐轮排除 → 非确诊报告（Mock 驱动）
4. `DeepSeekLlm` 构造合规请求（tool_choice=required、4 工具、system 含「家庭医生」）并解析 tool_call
5. provider 失败 → 回退下一个 provider（trace 记录 deepseek 失败 + openrouter 成功）
6. `DeepSeekLlm` e2e：头晕胸闷多轮（scripted「会读工作区」的假模型）→ 报告
7. `DeepSeekLlm` + Guard：模型违规出报告 → 被拒后改为追问（验证 Guard→LLM 反馈闭环）
8. `POST /api/consult` HTTP 往返 + 参数校验 + 可观测性（trace 含 tool/snapshot）

## 不做什么

- 不在测试中真连 DeepSeek/OpenRouter（用注入的 `CompleteFn` 模拟）
- 不做 extract_facts（M3）：症状仍用知识库词表 naive 种子，故测试输入需含规范症状词
- 不做 Firecrawl / hypothesis 系列工具 / 知识库扩到 10 组（M3）
- 不落 PG（M3）：可观测性先用内存 `MemoryTracer`
- 不把新路由挂进遗留 `server/index.ts`（迁移留待后续，避免影响现网 Prisma 链路）
